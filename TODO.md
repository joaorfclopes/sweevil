## Booking double-booking prevention (unique DB constraint)
Current `date + slot` compound index (`bookingModel.js`) is a query index, not a uniqueness constraint. Two simultaneous booking requests for the same slot can both pass the application-level check and both write. Add a sparse unique index: `bookingSchema.index({ date: 1, slot: 1 }, { unique: true, partialFilterExpression: { status: { $ne: 'CANCELED' } } })` — this rejects duplicate confirmed/pending bookings for the same slot while allowing re-booking of cancelled slots. Requires a migration to drop the existing non-unique index if already applied. Handle the `E11000 duplicate key` error in the booking creation route and return 409.
